**Unknowns / Risks**
- LP memory units do not match vLLM exactly: vLLM admits in KV blocks, not per-token memory. A first LP implementation can model tokens, but final translation must still call `allocate_slots()` and handle `None`.
- For hybrid KV cache groups, one “free block” is shared pool capacity, but per-request block needs can vary by manager/spec. Exact LP memory coefficients need a helper around `coordinator.get_num_blocks_to_allocate(...)`, not a naive token count.
- Decode readiness under speculative decoding and async placeholders is more subtle than `num_computed_tokens >= num_prompt_tokens`; the scheduler’s real formula includes `num_tokens_with_spec` and `num_output_placeholders`.
- Waiting requests with KV transfer states can have `num_computed_tokens > 0`; LP state collection must either support or initially exclude/block them.
- Preempting requests already scheduled earlier in the same step requires undoing local schedule bookkeeping, as native code does for default preemption.
- Chunk sizing for multimodal encoder constraints and Mamba block alignment can shrink a requested LP chunk to zero.

**Proposed Decomposition**

1. `_collect_lp_state()`: snapshot `waiting`, `skipped_waiting`, `running`, request fields, KV block state, and budgets.
2. `_classify_request_action_space()`: derive prefill/decode/preempt eligibility per request using vLLM’s progress semantics.
3. `_estimate_kv_costs()`: block-aware first, with optional per-token approximation logged explicitly.
4. `_compute_utility_weights()`: alpha/beta/gamma from arrival time and policy constants.
5. `_build_lp()` / `_solve_lp()`: isolated SciPy path with fallback.
6. `_extract_fractionals()`: deterministic integral/fractional conversion.
7. `_translate_lp_actions()`: perform vLLM-compatible queue moves, `allocate_slots()`, `_preempt_request()`, output assembly.
8. `_validate_scheduler_output()`: mirror native invariants before returning `SchedulerOutput`.
9. `_log_lp_metrics()`: build/solve/extract/translate timings and fallback reason.
